## 축 1 — 압도적 바이럴 및 생태계 침투

성공적인 복구 하나하나를 외부로 향하는 성장 신호로 바꾸고, 도구의 포지션을 일회성
수리 도구에서 **팀 단위 테스트 대시보드**로 재정의합니다.

### 1.1 "Failed Test Registry" — 트렌드 분석 리포트
사내 사용 데이터를 집계하여 **어떤 컴포넌트·셀렉터 패턴이 가장 자주 깨지는지** 통계를
내고 마크다운 리포트로 출력합니다.

- **효과:** _"우리 팀은 텍스트 변경으로 인한 테스트 실패율이 40%다"_ 같은 정량적 분석을
  제공 → 단순 복구 툴을 넘어 **팀 내 테스트 관리 대시보드**로 포지셔닝.
- **선행:** 구조화된 복구 히스토리 데이터(4.2 참조).
- **제안: `0.6.0`**

### 1.2 GitHub Star 연동 "Free Token Event" 프로모션 아키텍처
개발자가 플레이그라운드/CLI에서 본인 GitHub 계정으로 Star를 누르면, 자체 프록시 서버를
경유해 **무료 복구 크레딧**(OpenAI/Anthropic)을 일정량 충전해 줍니다.

- **효과:** 스타 1K → 5K 점프를 위한 가장 확실한 오픈소스 마케팅 트리거.
- **주의:** 크레딧 프록시 백엔드가 필요 → 코어/래퍼 분리 원칙상 **CLI 코어 밖**에 위치.
- **제안: `1.0.0`+ (별도 백엔드 트랙)**

### 1.3 "Wall of Saved Hours" — 글로벌 명예의 전당 배지
도입한 모든 레포/기업이 아낀 개발자 공수를 합산하여 레포 메인에 **라이브 배지**로 노출.

- **효과:** _"이 툴 덕분에 전 세계 개발자가 N만 시간을 아꼈다"_ 는 대외 신뢰성과 거대한
  서사 형성.
- **선행:** 옵트인 텔레메트리 집계 + 호스팅 카운터.
- **제안: `0.9.0`**

### 1.4 "Engineering Case Study" 자동 생성기
대규모 테스트 세트를 자가 복구하는 데 성공했을 때, 그 과정을 _'AI가 어떻게 AST 분석을
통해 셀렉터 결함을 해결했는가'_ 에 대한 **기술 블로그 포스팅용 마크다운 초안**으로 자동
빌드합니다.

- **효과:** 사용 후 Velog·Medium 등에 글을 퍼다 나르게 만들어 기술적 바이럴을 자동화.
- **제안: `0.9.0`**

### 1.5 Slack / X(트위터) 봇 — 1분 요약 패치 알림
복구가 완료되면 **Slack 또는 X**로 "어떤 컴포넌트가 깨졌고 패치가 무엇을 바꿨는지"를
간결히 요약해 전송합니다.

- **효과:** 팀 공유 + 매 수정마다 도구를 소셜에 자연 노출.
- **제안: `0.6.0`**

## 축 2 — 엔터프라이즈 신뢰 · 보안 · 감사(Audit)

> _(재구성 축) — 원본의 2.x 항목 텍스트가 상당 부분 깨져 있어, 복원 가능한 의도는
> 규제 대상 기업을 위한 감사/컴플라이언스 추적 기능입니다._

### 2.1 불변(immutable) 복구 감사 추적 _(재구성)_
모든 복구 실행을 **영구 보존되는 위변조 방지 기록**(서명·구조화 로그 또는 내보내기 가능한
XML 등)으로 변환하여 무엇이·왜·어떤 모델로 바뀌었는지 남깁니다.

- **효과:** 대기업 화이트해커팀·보안팀이 **보안 규제 감사를 통과**하는 데 필수적인 명분을
  제공.
- **제안: `0.8.0`**

_(그 외 축 2의 2.2–2.x 항목은 원본이 불명확하여 추후 확인 전까지 보류합니다.)_

## 축 3 — 극한의 개발자 경험(DX) 및 툴링 체인 형성

첫 진입 마찰을 0으로 만들고, CLI에 익숙하지 않은 사용자까지 타겟을 확장합니다.

### 3.1 "Healer-Init" — 원클릭 프로젝트 진단 CLI
`npx e2e-healer init` 실행 시 현재 레포의 Playwright 설정·테스트 코드를 분석하여
_"이 레포는 AI 복구 성공률 92% 예상"_ 등의 리포트와 **기본 CI YAML을 자동 세팅**합니다.

- **효과:** 사용자가 첫 진입 단계에서 겪는 피로도를 제로로.
- **제안: `0.6.0`**

### 3.2 브라우저 익스텐션 "Live DOM Picker"
크롬 익스텐션으로 로컬에 띄운 화면을 보며 "이 영역 깨진 거 고쳐줘"라고 타겟팅하면 CLI와
연동되어 **해당 컴포넌트만 핀포인트 복구**합니다.

- **효과:** CLI가 익숙하지 않은 주니어·QA 엔지니어까지 유저 타겟 확장.
- **주의:** 익스텐션 + CLI 브릿지 필요 → 별도 전달 표면.
- **제안: `1.0.0`+ (별도 익스텐션 트랙)**

### 3.3 아키텍처 경계 규칙 레이어 _(재구성)_
복구가 **내부 아키텍처 경계**(예: Entities / 도메인 레이어) 안에 머물도록 **제한하는 규칙
레이어** — 패치가 모듈 경계를 넘나들지 않도록 보장.

- **효과:** 대형 구조를 가진 모던 프론트엔드 팀이 **코드가 지저분해질 걱정 없이** 도입.
- **제안: `0.7.0`**

## 축 4 — AI 에이전트 연산 고도화 및 비용 최적화

코어 엔진을 더 싸게·빠르게·범용적으로 만듭니다.

### 4.1 Tree-Sitter 기반 "Semantic Code Chunking"
0.3.0에서 도입한 tree-sitter를 고도화 — 파일 전체를 LLM에 넘기는 대신 에러가 발생한 라인의
**상위 부모 블록(JSX Element 단위)만** 떼어 컨텍스트로 주입.

- **효과:** 토큰 소모량 **최대 80% 절감** — 속도와 비용 동시 확보.
- **제안: `0.5.0`**

### 4.2 "Past Healing History" — RAG / 파인튜닝 메모리 레이어
이전에 성공적으로 고친 셀렉터 변경 패턴(예: `data-testid` 명명 규칙 변경)을 **벡터 DB 또는
로컬 JSON**에 기억해 두고, 유사 에러 발생 시 **LLM 호출 없이 즉시 1차 패치**를 시도.

- **효과:** 반복되는 실패 패턴에 대한 LLM 비용 제거 — 1.1 레지스트리의 토대.
- **제안: `0.5.0`**

### 4.3 프레임워크 적응형 프롬프트 레이어 _(재구성)_
프레임워크 사양(React 19, Vue 3, Svelte …)에 따라 **최적의 복구 셀렉터 스타일을 동적으로
조절**하는 상위 프롬프트 엔지니어링 레이어.

- **효과:** 에이전트 범용성 극대화 — 어떤 프론트엔드 스택 유저가 유입돼도 높은 복구 성공률.
- **제안: `0.7.0`**

## 릴리스 계획

제안 순서와 원칙: **코어 엔진 개선을 먼저**(효과가 누적되고 외부 인프라 불필요),
**로컬 DX·리포팅을 그다음**, **서사·마케팅은 이야기할 데이터가 쌓인 뒤**, **호스팅 백엔드가
필요한 것은 `1.0.0` 이후**로 배치하여 "CLI 코어 / 얇은 래퍼" 분리 원칙을 지킵니다.

| 버전      | 테마                     | 항목                                                   |
|-----------|--------------------------|--------------------------------------------------------|
| **0.5.0** | 엔진 비용·메모리         | 4.1 Semantic Chunking, 4.2 복구 히스토리 메모리        |
| **0.6.0** | 리포팅 · 무마찰 DX       | 1.1 Failed Test Registry, 1.5 Slack/X 봇, 3.1 Healer-Init |
| **0.7.0** | 범용성 · 코드 위생       | 4.3 프레임워크 적응형 프롬프트, 3.3 경계 규칙 레이어    |
| **0.8.0** | 엔터프라이즈 · 컴플라이언스 | 2.1 불변 감사 추적                                    |
| **0.9.0** | 서사 · 바이럴            | 1.4 Case-Study 생성기, 1.3 Wall of Saved Hours         |
| **1.0.0** | 안정화 + 성장 백엔드     | 안정화; 1.2 Free-Token 프록시·3.2 Live DOM Picker 착수 |
| **1.0.0+**| 별도 트랙 (호스팅 인프라)| 1.2 Free-Token 이벤트, 3.2 브라우저 익스텐션           |

### 이 순서인 이유

- **0.5.0 먼저 (4.1 + 4.2):** 둘 다 0.3.0의 tree-sitter 토대를 직접 확장하는 순수 코어 엔진
  개선. 비용·지연을 낮추는 동시에, 이후 1.1 레지스트리와 1.3 절감 시간 카운터가 의존하는
  **구조화된 복구 히스토리 데이터**를 만들어 냅니다. **데이터 레이어를 그것을 읽는 기능보다
  먼저** 출시합니다.
- **0.6.0 (1.1 + 1.5 + 3.1):** 전부 로컬, 백엔드 불필요. 도구를 **대시보드 + 온보딩 경험**으로
  전환하고 첫 바이럴 표면(Slack/X)을 만듭니다 — 팀 가치가 가장 빠르게 눈에 보이는 경로.
- **0.7.0 (4.3 + 3.3):** 시장 확장 — 프레임워크 적응형으로 모든 프론트 스택을 흡수하고,
  경계 규칙으로 대형 엔터프라이즈 코드베이스의 도입 장벽을 해소.
- **0.8.0 (2.1):** 컴플라이언스는 엔터프라이즈 도입의 관문 — 도구가 이미 감사할 가치가 있을
  만큼 유용해진 뒤에 배치.
- **0.9.0 (1.4 + 1.3):** 실제 사용 데이터(0.5–0.6)가 쌓인 **뒤에야** 설득력을 갖는 마케팅·서사
  기능.
- **1.0.0+ (1.2, 3.2):** **호스팅 인프라**(크레딧 프록시, 브라우저 익스텐션)가 필요하며 CLI
  코어 밖에 있어야 합니다. 코어를 로컬 실행·CI 호출 가능한 상태로 유지하기 위해 **별도
  레포/트랙**으로 취급합니다.
